## ОСС (Общее собрание собственников)

Модуль автоматизирует полный цикл заочного/очно-заочного голосования собственников
ТСН: формирование индивидуальных бланков решений, рассылку их по email, печать,
загрузку заполненных ответов и подсчёт итогов по кворуму.

Голосование строится вокруг сущности **`Decision`** — это персональный бланк решения
одного собственника по всем вопросам повестки.

### Доменная модель

#### `Decision` (`model/entity/Decision.kt`, таблица `decision`)

Один бланк решения = один собственник.

| Поле | Тип | Назначение |
|------|-----|-----------|
| `id` | `Long` | PK. Используется как **ID бланка** в Excel-шаблоне и при загрузке ответов |
| `uuid` | `String` (unique) | Технический ключ дедупликации: `"<rooms> <square> <percentage>"` |
| `fullName` | `String` | ФИО собственника (из `OwnerEntity.fullName`) |
| `square` | `BigDecimal` | Суммарная площадь всех помещений собственника, м² |
| `percentage` | `BigDecimal` | Суммарная доля собственника в общем имуществе, % |
| `rooms` | `Set<Long>` (jsonb) | ID помещений собственника |
| `numbersOfRooms` | `String` | Человекочитаемый список помещений, напр. `кв.012, мм.034` |
| `emails` | `Set<String>` (jsonb) | Адреса для рассылки бланка |
| `notified` | `Boolean` | Бланк успешно отправлен на email |
| `dateOfNotified` | `LocalDateTime?` | Когда отправлен |
| `voted` | `Boolean` | Ответы загружены (собственник проголосовал) |
| `print` | `Boolean` | Пометка «печатать бумажный бланк» (выставляется вручную в БД) |
| `blank` | `TEXT` | Готовый текст бланка (шапка + помещения + подвал), строки через `\n` |
| `createDate` | `LocalDateTime` | Когда сформирован бланк |
| `answers` | `List<AnswerEnum>?` (jsonb) | Ответы по вопросам повестки в порядке вопросов |

#### `AnswerEnum` (`enums/AnswerEnum.kt`)

Ответ по одному вопросу. Парсер распознаёт как слова, так и цифры:

| Значение | Распознаётся из | В отчёте |
|----------|-----------------|----------|
| `YES` | `«За»`, `1` | `З` |
| `NO` | `«Против»`, `0` | `П` |
| `UNDEFINED` | `«Воздержался»`, `2`, **и всё нераспознанное** | `В` |

> ⚠️ `determine()` возвращает `UNDEFINED` для любого нераспознанного значения —
> опечатки в ответах молча трактуются как «воздержался».

`Decision` формируется из **`OwnerEntity`** (собственники) и **`Room`** (помещения):
площадь и доля берутся из помещений, привязанных к собственнику.

### Жизненный цикл голосования

```
1. PrepareDecision   POST /api/decisions                 → создаёт бланки из собственников/помещений
2. Печать шаблона    GET  /api/reports/decisions/templates/decisions  → Excel-шаблон для ввода ответов
3. Рассылка          POST /api/decisions/send            → отправляет .docx бланки на email
   (опц.) Печать     GET  /api/reports/decisions/not-voted | /selected-to-print → .docx файлы
4. Сбор ответов      POST /api/files/answers/importer     → загрузка заполненного Excel, voted=true
5. Итоги             GET  /api/reports/decisions          → Excel со сводкой и кворумом
```

Подробно по шагам:

1. **Подготовка бланков** — система обходит всех собственников, считает суммарную
   площадь/долю по их помещениям и генерирует персональный текст бланка из
   шаблона (`Template` id=5: `header` + перечень помещений + `footer`).
2. **Шаблон для ответов** — выгружается Excel с листом `«Ответы на вопросы»`,
   где каждая строка = бланк (`ID`, `ФИО`, недвижимость, площадь, доля) и пустые
   колонки `Вопрос 1..N`. Этот же файл потом заполняется и загружается обратно.
3. **Рассылка/печать** — `.docx` бланки рассылаются по email собственника
   (шаблон письма `Template` id=6) и/или печатаются в файлы.
4. **Загрузка ответов** — заполненный Excel загружается, ответы сопоставляются
   с бланками по `ID`, у бланка проставляется `voted=true` и сохраняются `answers`.
5. **Итоговый отчёт** — Excel с 4 листами: общая информация, результаты по
   вопросам с проверкой кворума, проголосовавшие, не проголосовавшие.

### REST API

Все эндпоинты под контекстом `/api`. Требуется JWT (Bearer). Swagger UI:
`/api/swagger-ui/index.html`.

#### DecisionController — `/api/decisions` (`rest/decision/DecisionController.kt`)

**`POST /api/decisions` — подготовить бланки.** Формирует бланки `Decision` по всем
собственникам. Использует `Template` **id=5** (`header`/`footer`). Сохраняет все
бланки в БД.

```json
// DecisionResponse
{ "totalSize": 250, "square": 18345.60, "percentage": 100.00 }
```

> ⚠️ Дедупликация пока **не реализована** (`// TODO to add skip duplicate` в
> `DecisionService.prepareDecision`). Повторный вызов создаст дубликаты бланков —
> запускать на чистой таблице `decision` либо предварительно её очищать
> (см. [Повторная загрузка / перезапуск голосования](#повторная-загрузка--перезапуск-голосования)).

**`POST /api/decisions/send` — разослать бланки на email.** Берёт
**непроголосовавшие** бланки, для каждого генерирует `.docx` из поля `blank`, тело
письма — из `Template` **id=6** (плейсхолдер `{{fullName}}` заменяется на ФИО).
Отправляет на все `emails` бланка, при успехе выставляет `notified=true`.

- Тема: `ТСН МР17дом1. Решение собственника <ФИО>. (<помещения>)`
- Вложение: `Решение собственника (<помещения>) <ФИО>.docx`

```json
// MailingResponse
{ "totalDecisions": 250, "totalEmails": 180, "sentDecisions": 120, "sentEmail": 175 }
```

> Отправка идёт синхронно в цикле — на больших рассылках запрос выполняется долго.

#### DecisionReportController — `/api/reports/decisions` (`rest/decision/DecisionReportController.kt`)

**`GET /api/reports/decisions` — итоговый Excel** (`Decisions_<timestamp>.xlsx`).
Сводка по всем бланкам, листы:

| Лист | Содержимое |
|------|-----------|
| `Основная информация` | площадь, доли, сколько проголосовало/нет |
| `Результаты` | по каждому вопросу: «За/Против/Воздержался» (доля и %), пройден ли кворум |
| `Проголосовали` | список с расшифровкой ответов (`З/П/В`) |
| `Не проголосовали` | список, отсортированный по доле (по убыванию) |

**`GET /api/reports/decisions/templates/decisions` — Excel-шаблон ответов**
(`Template_<timestamp>.xlsx`). Лист `«Ответы на вопросы»`: по строке на бланк
(`№`, `ID`, `ФИО`, `Недвижимость`, `Площадь`, `Доля`) и пустые колонки `Вопрос 1..18`.
Заполняется ответами и загружается через импортёр.

> Колонка `ID` обязательна — по ней ответы сопоставляются с бланками при импорте.

**`GET /api/reports/decisions/not-voted` — печать непроголосовавших.** Генерирует
`.docx` бланки для всех непроголосовавших в каталог `etc/not-voted/`. Возвращает
количество файлов.

**`GET /api/reports/decisions/selected-to-print` — печать отмеченных.** То же, но
только для бланков с `print=true` (флаг ставится вручную в БД). Каталог
`etc/selected-to-print/`.

#### Импорт ответов — `/api/files/answers/importer` (`rest/FileImporterController.kt`)

`multipart/form-data`, поле `file` — заполненный Excel (лист `«Ответы на вопросы»`).
Проверяет расширение и дубликат файла (SHA-256), парсит ответы
(`AnswerParser` → `DecisionService.parseAndSave`), по `ID` находит бланк, пишет
`answers` и `voted=true`. Метаданные файла сохраняются с типом
`FileTypeEnum.DECISION_ANSWERS`.

```json
// AnswerInfoResponse
{ "fileName": "answers.xlsx", "totalSize": 120 }
```

> Строка пропускается, если все ячейки вопросов пустые. Парсер останавливает
> чтение вопросов строки на первой пустой ячейке — частично заполненные строки
> учтут только идущие подряд первые ответы.

### Повторная загрузка / перезапуск голосования

При перезаливке ответов или повторном проведении нужно вручную сбросить состояние
в БД (дедупликация бланков и импорта файлов работает по этим данным):

1. Сбросить ответы по всем бланкам:
   ```sql
   update decision set answers = null, voted = false;
   ```
2. Удалить из БД запись ранее загруженного файла ответов (иначе сработает
   дедупликация по имени/контрольной сумме):
   ```sql
   delete from file where name = 'Template_2024-04-25-01-28-24.xlsx';
   ```
3. Загрузить новый файл ответов: `POST /api/files/answers/importer`
4. Скачать итог: `GET /api/reports/decisions`

### Шаблоны (Template)

Сущность `Template` (`model/entity/Template.kt`, таблица `template`) хранит тексты.
Модуль голосования использует фиксированные ID (захардкожены в `DecisionService`):

| ID | Где используется | Поля |
|----|------------------|------|
| **5** | `prepareDecision()` — текст бланка | `header` (шапка бланка), `footer` (подвал/вопросы) |
| **6** | `sendDecisions()` — тело письма | `body` с плейсхолдером `{{fullName}}` |

> ⚠️ ID шаблонов зашиты в коде. Перед проведением ОСС убедитесь, что в таблице
> `template` есть записи с id=5 и id=6 и корректным содержимым (текущая повестка
> в `footer` шаблона id=5).

### Подсчёт результатов и кворум

Логика в `excel/Decisions.kt` (`createPercentageSheet`) и `excel/QuestionResult.kt`.

- По каждому вопросу суммируется **доля собственности** (`percentage`), а не число
  голосов.
- Кворум: вопрос **пройден**, если доля «За» превышает `QUORUM = 67.0`
  (`utils/Constants.kt`). Иначе показывается, сколько процентов не хватило.
- Количество вопросов повестки: `NUMBER_OF_QUESTIONS = 18` (`utils/Constants.kt`).
- Подписи вопросов захардкожены в `descriptionOfQuestions` внутри
  `createPercentageSheet` — это повестка **конкретного прошедшего ОСС**, при новом
  собрании их нужно обновить.

### Конфигурация и допущения

Перед новым ОСС проверьте/обновите захардкоженные значения:

1. **`NUMBER_OF_QUESTIONS`** (`utils/Constants.kt`) — число вопросов повестки.
2. **`QUORUM`** (`utils/Constants.kt`) — порог прохождения вопроса (сейчас 67%).
3. **`descriptionOfQuestions`** в `excel/Decisions.kt` — подписи вопросов в отчёте.
4. **Шаблоны id=5 и id=6** в таблице `template` — текст бланка и письма.
5. **Структура колонок** Excel жёстко согласована между генерацией и парсингом:
   - шаблон (`excel/Answer.kt`): `0:№`, `1:ID`, `2:ФИО`, `3:Недвижимость`,
     `4:Площадь`, `5:Доля`, далее `6+:Вопрос N`;
   - парсер (`parser/AnswerParser.kt`): `id` из колонки **1**, `fullName` из **2**,
     вопросы с колонки **6**; пропускается 1 строка заголовка;
   - имя листа должно быть ровно `«Ответы на вопросы»`.
6. **Каталоги вывода** `.docx`: `etc/not-voted/`, `etc/selected-to-print/`,
   вложения писем — `etc/blanks/`. Должны существовать/быть доступны на запись.
7. **Дедупликация бланков отсутствует** — `prepareDecision` запускать на пустой
   таблице `decision`.
